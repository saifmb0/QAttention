% ABSTRACT
% =====================================================================
\begin{abstract}
Tree-structured speculative decoding---employed by
EAGLE-2/3~\cite{li2024eagle2,li2025eagle3},
Medusa~\cite{cai2024medusa}, and Sequoia~\cite{chen2024sequoia}---verifies
hundreds to thousands of draft tokens per step.
During verification, each token attends only to its
$\depth{+}1$ ancestors in the tree, yet production attention kernels
(FlashAttention-2~\cite{dao2023flashattention2},
FlashInfer~\cite{ye2025flashinfer}, DeFT~\cite{yao2024deft})
perform $\bigO{\numtokens^2}$ work, oblivious to this sparsity.

We present \textbf{\method{}}, a Triton kernel that exploits the
BFS ancestor structure to reduce per-head complexity from
$\bigO{\numtokens^2}$ to $\bigO{\numtokens \cdot \depth}$.
The kernel replaces the KV-block loop of FlashAttention-2
with an ancestor walk of exactly $\depth{+}1$ scattered gathers
per Q-tile, eliminating the need for any mask materialisation.

We evaluate across two complementary benchmarks on an NVIDIA H100 SXM:
(i)~a \emph{SOTA comparison} against FlashAttention-2, FlashInfer, and DeFT
at complete $\branching$-ary trees ($\branching \in \{2,3,4\}$, $\depth \leq 8$),
where \method{} achieves up to
\textbf{49$\boldsymbol\times$} over FlashAttention-2,
\textbf{309$\boldsymbol\times$} over FlashInfer, and
\textbf{114$\boldsymbol\times$} over DeFT; and
(ii)~a \emph{micro-benchmark} at EAGLE-3-class configurations
($\branching \in \{8\text{--}20\}$, $\depth \leq 32$, $\batchsize \leq 128$,
$\headdim=128$, $\nheads=32$) with a \emph{simulated prefix KV cache}
($\prefixlen \in \{0, 1024, 4096\}$).

Our main findings:
\textbf{(1)}~With no prefix ($\prefixlen=0$),
\method{} wins 92\% of 350 configurations (median 3.1$\times$, peak 8.9$\times$);
\textbf{(2)}~With a realistic prefix ($\prefixlen=1024$),
\method{} wins \emph{100\% of configurations}, including at EAGLE-3's
default operating point ($\branching=10$, $\depth=7$): 1.76--2.82$\times$
across batch sizes;
\textbf{(3)}~At $\prefixlen=4096$, the advantage grows further
(median 4.3$\times$, worst-case 1.83$\times$), because the dense
baseline materialises an $[\batchsize, \nheads, \numtokens, \prefixlen{+}\numtokens]$
weight matrix while \method{} delegates the prefix to flash attention
and handles only the sparse tree part.

At small tree sizes ($\numtokens < 50$, $\prefixlen=0$), kernel launch
overhead dominates and dense attention remains competitive.
All code, data, and reproduction scripts are open-sourced.
\end{abstract}

\section{Introduction}\label{sec:intro}
% =====================================================================

Autoregressive decoding in large language models (LLMs) is
fundamentally memory-bound: each token generation requires a full
forward pass whose arithmetic intensity is limited by the
key--value (KV) cache read~\cite{leviathan2023fast, chen2023accelerating}.
Speculative decoding addresses this bottleneck by generating candidate
tokens cheaply with a \emph{draft model} and verifying them in
parallel with the \emph{target model}~\cite{leviathan2023fast,
chen2023accelerating}.

Recent methods---EAGLE-2/3~\cite{li2024eagle2,li2025eagle3},
Medusa~\cite{cai2024medusa}, Sequoia~\cite{chen2024sequoia}, and
SpecInfer~\cite{miao2024specinfer}---organise draft tokens into
\emph{trees}, increasing acceptance rates by exploring multiple
continuations simultaneously.
Tree-structured verification generates trees with branching factors
$\branching = 2\text{--}20$ and depths $\depth = 3\text{--}32$,
yielding $\numtokens = 30\text{--}500{+}$ draft tokens per step.

The critical observation is that during tree-structured verification,
each draft token $i$ attends only to the $\depth_i + 1$ tokens on its
root-to-leaf path---its \emph{ancestors} in the BFS-ordered tree.
A typical EAGLE-3 tree ($\branching = 10$, $\depth = 7$, $\numtokens \approx 60$)
has only $\approx 8$ non-zero entries per row of the attention mask,
yet standard dense attention computes all 60 dot products.
This is $>87\%$ wasted computation on the tree part alone.

When a prefix KV cache of length $\prefixlen$ is present---the
common case during generation, where $\prefixlen = 1024\text{--}8192$
committed tokens precede the draft tree---the waste is \emph{worse}.
EAGLE-3's implementation~\cite{li2025eagle3} materialises the full
$[\batchsize, \nheads, \numtokens, \prefixlen {+} \numtokens]$ attention
weight matrix via \texttt{torch.matmul}, even though the
$[\numtokens \times \numtokens]$ tree quadrant is $>87\%$ sparse.

\paragraph{Contributions.}
\begin{enumerate}[nosep,leftmargin=*]
\item \textbf{Algorithm.}
    We present an $\bigO{\numtokens \cdot \depth}$ ancestor-sparse
    flash attention algorithm that replaces the KV-block loop
    with an ancestor walk of $\depth + 1$ scattered gathers.
    No attention mask is materialised; the tree topology is
    computed analytically from the BFS parent formula
    $\parent(k) = \lfloor (k{-}1)/\branching \rfloor$.

\item \textbf{Kernel.}
    We implement the algorithm as an autotuned Triton kernel with
    specialised configurations for four GPU generations
    (Turing SM75, Ada SM89, Hopper SM90, Blackwell SM120).
    Three variants are provided: standard, LSE-returning (for
    prefix merging), and paged-KV (for vLLM compatibility).

\item \textbf{Prefix--tree split architecture.}
    For the realistic setting where draft tokens attend to both a
    prefix KV cache and the tree, we propose a
    \emph{flash prefix + ragged tree + online-softmax merge}
    pipeline that avoids materialising any large attention matrix.

\item \textbf{Comprehensive evaluation.}
    We evaluate on 1{,}050 micro-benchmark configurations
    (5 batch sizes $\times$ 7 branching factors $\times$ 10 depths
    $\times$ 3 prefix lengths) plus 192 SOTA comparison configurations,
    including correctness verification and roofline profiling.
\end{enumerate}

\section{Background}\label{sec:background}
% =====================================================================

\subsection{Speculative Decoding}

Speculative decoding~\cite{leviathan2023fast,chen2023accelerating}
uses a lightweight draft model to propose $\gamma$ candidate tokens,
which the target model verifies in a single batched forward pass.
Accepted tokens advance the sequence; rejected tokens trigger
resampling.
The expected speedup depends on the acceptance rate $\alpha$
and the draft/target cost ratio.

\subsection{Tree-Structured Drafts}

Methods like EAGLE-2/3~\cite{li2024eagle2,li2025eagle3},
Medusa~\cite{cai2024medusa}, and Sequoia~\cite{chen2024sequoia}
extend speculative decoding by organising candidates into $\branching$-ary
trees of depth $\depth$.
The tree is laid out in breadth-first order: root at position 0,
children of node $k$ at positions $\branching k + 1, \ldots, \branching k + \branching$.
The number of nodes in a complete tree is
$\numtokens = (\branching^{\depth+1} - 1) / (\branching - 1)$,
though practical systems (e.g., EAGLE-3) prune this to a budget
of $\numtokens \approx \lceil 6\branching\depth / 7 \rceil$
tokens~\cite{li2025eagle3}.

\paragraph{Attention rule.}
During tree verification, the attention mask for the tree portion is
\emph{not} lower-triangular: token $i$ attends to token $j$ if and
only if $j$ is an ancestor of $i$ in the tree (including $i$ itself).
A token at depth $\depth_i$ therefore has exactly $\depth_i + 1$ attended keys,
compared to the $\numtokens$ keys a dense kernel computes.

\subsection{Dense Attention Waste on Trees}\label{sec:waste}

For the tree-only attention block (ignoring prefix), the fraction of
\emph{useful} FLOPs when using a dense $\bigO{\numtokens^2}$ kernel is:
\begin{equation}\label{eq:util}
  \eta_{\text{tree}} = \frac{\sum_{i=0}^{\numtokens-1} (\depth_i + 1)}
                             {\numtokens^2}.
\end{equation}
For a complete $\branching$-ary tree of depth $\depth$,
$\eta_{\text{tree}} \to 0$ as $\branching$ or $\depth$ grows.
At $\branching = 10$, $\depth = 7$: $\eta_{\text{tree}} < 13\%$.
At $\branching = 4$, $\depth = 6$ ($\numtokens = 5{,}461$):
$\eta_{\text{tree}} < 0.13\%$.

\paragraph{Prefix amplification.}
When a prefix cache of length $\prefixlen$ is present, the target model's
attention layer operates on KV of length $\prefixlen + \numtokens$.
EAGLE-3's implementation~\cite{li2025eagle3} materialises the full
$[\numtokens \times (\prefixlen + \numtokens)]$ weight matrix via
\texttt{torch.matmul}.
While every tree token legitimately attends to all $\prefixlen$ prefix
tokens (dense), the $[\numtokens \times \numtokens]$ tree-to-tree quadrant
remains almost entirely masked.
The wasted memory scales as
$\batchsize \cdot \nheads \cdot \numtokens \cdot (\prefixlen + \numtokens) \cdot 2$ bytes (fp16),
which at $\batchsize = 128$, $\nheads = 32$, $\numtokens = 60$, $\prefixlen = 4096$
is $> 3$ GB for a \emph{single attention layer}.

\section{Method}\label{sec:method}
% =====================================================================

\subsection{Ancestor-Sparse Flash Attention}\label{sec:algo}

FlashAttention-2~\cite{dao2023flashattention2} tiles Q into blocks
of $\text{BLOCK\_M}$ rows and iterates over $\lceil \numtokens / \text{BLOCK\_N} \rceil$
KV blocks.
Our key insight is that \textbf{every query row needs exactly
$\depth + 1$ key--value pairs}---its ancestors.
We replace the KV-block loop with an \emph{ancestor walk}:

\begin{equation}\label{eq:ancestor}
  a^{(0)}(i) = i, \qquad
  a^{(s)}(i) = \left\lfloor \frac{a^{(s-1)}(i) - 1}{\branching} \right\rfloor
  \quad \text{for } s = 1, \ldots, \depth.
\end{equation}

At each step $s$, every query row $q_m$ in the tile loads the single
K and V vector at position $a^{(s)}(q_m)$ via \emph{scattered gather}.
Different rows within a tile load from different positions---this
is fundamentally different from FlashAttention-2, where all rows
share the same KV block.

\paragraph{Kernel structure.}
Each Triton program handles one (sequence, head) pair and one
Q-tile of $\text{BLOCK\_M}$ rows.
The inner loop runs exactly $\depth + 1$ steps:

\begin{enumerate}[nosep,leftmargin=*]
\item \textbf{Ancestor index:} Compute $a^{(s)}(q_m)$ for each row $m$ via
      Eq.~\ref{eq:ancestor}. Global offset:
      $\text{seq\_start} + a^{(s)}(q_m)$.
\item \textbf{Scattered K/V load:} Gather $K[a^{(s)}(q_m), :]$ and
      $V[a^{(s)}(q_m), :]$ per row (each row loads from a potentially
      different address).
\item \textbf{Score \& online softmax:} Compute $s_m = q_m \cdot k_m / \sqrt{\headdim}$.
      A duplicate guard sets $s_m = -\infty$ when $a^{(s)} = a^{(s-1)}$
      (query reached root before step $\depth$).
      Update running max $m_i$, scale factor $\ell_i$, and accumulator
      using the standard online softmax~\cite{dao2023flashattention2}.
\item \textbf{Accumulate:} $\text{acc}_m \mathrel{+}= \exp(s_m - m_{\text{new}}) \cdot v_m$.
\end{enumerate}

\paragraph{Complexity.}
Each Q-tile of $\text{BLOCK\_M}$ rows performs $\depth + 1$ dot products
of dimension $\headdim$, giving $\bigO{\numtokens \cdot \depth \cdot \headdim}$
work per head---versus $\bigO{\numtokens^2 \cdot \headdim}$ for dense.
The work reduction factor is $\numtokens / (\depth + 1)$, which
at $\branching = 10$, $\depth = 7$, $\numtokens = 60$ is $7.5\times$,
and at $\branching = 4$, $\depth = 6$, $\numtokens = 5{,}461$ is
$780\times$.


\subsection{Prefix--Tree Split Architecture}\label{sec:prefix}

In the realistic serving scenario, each of the $\numtokens$ draft tokens
attends to $\prefixlen$ committed prefix tokens (dense) \emph{plus}
its $\depth_i + 1$ tree ancestors (sparse).
Rather than materialising the full
$[\numtokens \times (\prefixlen + \numtokens)]$ weight matrix, we
decompose the attention into two independent blocks and merge via
online softmax:

\begin{enumerate}[nosep,leftmargin=*]
\item \textbf{Prefix block:}
    Run FlashAttention-2 (via \texttt{\_scaled\_dot\_product\_flash\_attention})
    on $Q[\batchsize, \nheads, \numtokens, \headdim]$ and
    $K_{\text{pre}}[\batchsize, \nheads, \prefixlen, \headdim]$.
    This returns both the output $O_{\text{pre}}$ and the per-token
    log-sum-exp $\text{LSE}_{\text{pre}}$.

\item \textbf{Tree block:}
    Run \method{} on the tree KV only, using the LSE-returning variant
    to obtain $O_{\text{tree}}$ and $\text{LSE}_{\text{tree}}$.

\item \textbf{Online-softmax merge:}
    Combine the two blocks:
    \begin{align}\label{eq:merge}
      m &= \max(\text{LSE}_{\text{pre}},\; \text{LSE}_{\text{tree}}), \\
      w_p &= \exp(\text{LSE}_{\text{pre}} - m), \quad
      w_t = \exp(\text{LSE}_{\text{tree}} - m), \nonumber \\
      O &= \frac{w_p \cdot O_{\text{pre}} + w_t \cdot O_{\text{tree}}}{w_p + w_t}.
      \nonumber
    \end{align}
\end{enumerate}

This architecture has two advantages.
First, the prefix attention is handled by a highly optimised
flash kernel, leveraging its full tiling and software pipelining.
Second, the tree block processes only the sparse
$\numtokens \times \numtokens$ quadrant via the ragged kernel---no
$\prefixlen$-dependent memory allocation.
The total memory for the tree block is $\bigO{\batchsize \cdot \numtokens
\cdot \nheads \cdot \headdim}$, independent of prefix length.


\subsection{Ragged Batching}\label{sec:ragged}

Speculative decoding produces variable-length sequences across
requests: each request's tree may differ in depth, branching factor,
or pruning.
\method{} uses a \emph{ragged packed layout}: all $\batchsize$
sequences are concatenated along the token dimension, with a
\texttt{cu\_seqlens} offset array ($\batchsize + 1$ entries)
indexing into the packed tensor.
This eliminates padding waste entirely.

The kernel is autotuned for four GPU generations:

All tiers use \texttt{num\_stages=1}: the ancestor walk produces
data-dependent addresses, so software pipelining cannot help.
The autotune key is \texttt{[HEAD\_DIM, BRANCHING\_FACTOR, MAX\_DEPTH]}.

\paragraph{Hopper-specific optimisations.}
\begin{itemize}[nosep,leftmargin=*]
\item \textbf{Large Q-tiles.}
  Hopper's 228\,KB shared memory allows BLOCK\_M up to 512,
  improving Q reuse across ancestor steps.
\item \textbf{int64 pointer arithmetic.}
  At $\batchsize = 128$, $\branching = 4$, $\depth = 8$,
  the packed sequence contains $> 5.6$M tokens.
  Byte offsets exceed \texttt{INT32\_MAX};
  all offsets are computed in \texttt{tl.int64}.
\item \textbf{Parent-clamp-before-divide.}
  $\texttt{tl.maximum}(\text{cur} - 1, 0)$ prevents negative dividends
  in the ancestor formula, avoiding PTX lowering artefacts.
\end{itemize}

\section{Experimental Setup}\label{sec:setup}
% =====================================================================

\paragraph{Hardware.}
All experiments run on a single NVIDIA H100 SXM 80\,GB
(Hopper, SM 9.0, 3.35\,TB/s HBM3).

\paragraph{Software.}
PyTorch 2.6, Triton 3.2, CUDA 12.8, FlashInfer 0.2.2,
DeFT (Triton reimplementation).

\paragraph{Evaluation protocol.}
We use CUDA events with 10 warmup and 50 timed iterations,
reporting the \emph{median} latency.

\subsection{Benchmark Suite}

\paragraph{Benchmark 1: SOTA comparison (\Cref{sec:sota}).}
Compares \method{} against FlashAttention-2
(PyTorch SDPA flash backend), FlashInfer, and DeFT at \emph{complete}
$\branching$-ary trees
($\branching \in \{2,3,4\}$, $\depth \in \{1..8\}$,
$\batchsize \in \{1..128\}$, $\headdim = 64$, $\nheads = 8$).
192 configurations total.
These are tree-only ($\prefixlen = 0$) with the exact parameters
used in FlashInfer/DeFT codebases.

\paragraph{Benchmark 2: Micro-benchmark (\Cref{sec:micro}).}
Compares \method{} against EAGLE-3's \emph{actual attention
implementation}---the manual \texttt{torch.matmul + softmax + matmul}
path in \texttt{modeling\_llama\_kv.py}---at LLaMA-3.1-8B
dimensions ($\headdim = 128$, $\nheads = 32$):
$\batchsize \in \{8, 16, 32, 64, 128\}$,
$\branching \in \{8, 10, 12, 14, 16, 18, 20\}$,
$\depth \in \{5, 7, 9, 12, 14, 16, 20, 24, 28, 32\}$.
This benchmark adds a \emph{simulated prefix KV cache}
$\prefixlen \in \{0, 1024, 4096\}$.
1{,}050 configurations (939 valid; 111 Eagle OOM).

\paragraph{Correctness.}
104 configurations verified against PyTorch SDPA math-backend
reference with explicit ancestor mask.
Peak absolute error $< 2 \times 10^{-3}$ (fp16);
3 configurations skipped due to OOM at $\numtokens = 21{,}845$.

\paragraph{Profiling.}
36 configurations profiled with a roofline model.

\section{Results: SOTA Comparison}\label{sec:sota}
% =====================================================================

The SOTA benchmark compares \method{} against three baselines
at complete $\branching$-ary trees on the H100 SXM.
\Cref{tab:sota} reports representative configurations.

\subsection{vs.\ FlashAttention-2 (SDPA Flash)}

SDPA flash is the strongest baseline at small trees.
At $\numtokens \leq 85$ ($\branching{=}4$, $\depth{=}3$),
it is 1.4--2.7$\times$ \emph{faster}: its highly optimised
closed-source kernel amortises launch overhead over dense
computation that is still cheap at small $\numtokens$.

The crossover occurs around $\numtokens \approx 300\text{--}500$ tokens.
Beyond this, \method{}'s $\bigO{\numtokens \cdot \depth}$
scaling dominates:
$\branching{=}4$, $\depth{=}6$ ($\numtokens = 5{,}461$, $\batchsize{=}128$):
\textbf{49$\times$};
$\branching{=}3$, $\depth{=}7$ ($\numtokens = 3{,}280$, $\batchsize{=}128$):
\textbf{22.9$\times$}.

Out of 168 valid configurations, \method{} wins 45 (27\%).
The 73\% where SDPA wins are \emph{all} at $\numtokens \leq 364$.

\subsection{vs.\ FlashInfer}

FlashInfer's tree attention uses a TopK-sorted layout.
\method{} wins \textbf{85\% of 187 configurations} (median 7.0$\times$).
At $\branching{=}3$, $\depth{=}7$, $\batchsize{=}8$: 22.3$\times$;
at $\branching{=}4$, $\depth{=}7$, $\batchsize{=}32$: \textbf{309$\times$}.
FlashInfer wins only at $\batchsize{=}1$ with $\numtokens \leq 85$.

\subsection{vs.\ DeFT}

DeFT~\cite{yao2024deft} is a KV-guided sparse attention kernel.
\method{} wins \textbf{93\% of 189 configurations}
(median 23.3$\times$, peak 114$\times$).
DeFT's only advantage is at $\batchsize{=}1$, $\numtokens \leq 40$.

We attribute \method{}'s advantage over DeFT to workload balance:
DeFT distributes work per KV position, but the root token is attended
by all $\numtokens$ queries while leaf tokens by only 1.
\method{}'s Q-guided structure naturally balances work because
every query walks the same $\depth{+}1$ steps.

\section{Results: Micro-Benchmark}\label{sec:micro}

attention path used in EAGLE-3's
\texttt{modeling\_llama\_kv.py}~\cite{li2025eagle3}:
\begin{align*}
W &= \texttt{matmul}(Q, K^T) / \sqrt{\headdim} + \text{bias}, \\

where \text{bias} is 0 for ancestors and $-\infty$ elsewhere.
When $\prefixlen > 0$, this baseline materialises the full
$[\batchsize, \nheads, \numtokens, \prefixlen {+} \numtokens]$ tensor.
\method{} instead uses the flash prefix + ragged tree + LSE merge
pipeline (\Cref{sec:prefix}).

\paragraph{$\prefixlen = 1024$.}

\paragraph{$\prefixlen = 4096$.}
The advantage grows further: \textbf{100\% win rate},
worst case 1.83$\times$, median 4.27$\times$, peak 5.88$\times$.
At large batch sizes ($\batchsize \geq 64$), many Eagle configurations
\emph{OOM}---the $[\batchsize, \nheads, \numtokens, 4096 {+} \numtokens]$
tensor exceeds 80\,GB---while \method{} completes normally.
102 of 350 configurations resulted in Eagle OOM at $\prefixlen{=}4096$.

\paragraph{Why the speedup grows with $\prefixlen$.}
Eagle's cost scales as $\bigO{\numtokens \cdot (\prefixlen + \numtokens)}$:
it materialises the full weight matrix.
\method{}'s cost is
$\underbrace{\bigO{\numtokens \cdot \prefixlen / \text{tile}}}_{\text{flash prefix}}
+ \underbrace{\bigO{\numtokens \cdot \depth}}_{\text{ragged tree}}$,
where the flash prefix is \emph{tiled} (no materialisation) and the
tree part is independent of $\prefixlen$.

\section{Roofline Analysis}\label{sec:roofline}
% =====================================================================

We profile the ragged kernel across 36 configurations.
At typical tree sizes the kernel is solidly \emph{HBM-bound}:
arithmetic intensity is $I = \depth + 1$ FLOPs/byte, far below
the H100's balance point $I^* \approx 217$ FLOPs/byte.

At $\branching{=}4$, $\depth{=}5$, $\numtokens{=}1{,}365$, $\batchsize{=}4$:
the kernel achieves 1.08 TFLOP/s and 358.6\,GB/s---33.6\% of
peak HBM bandwidth after accounting for L2 hit rate on
ancestor-overlapping loads.
Utilisation improves with larger trees and batch sizes.

The memory-bound regime is \emph{favourable}: it means the kernel
is bottlenecked by the same resource (HBM bandwidth) as the rest
of inference, and further improvements would come from fusing
the prefix + tree into a single kernel launch to reduce
round-trips to HBM.

\section{End-to-End Evaluation}\label{sec:e2e}
% =====================================================================

We evaluate end-to-end token throughput using EAGLE-3~\cite{li2025eagle3}
with LLaMA-3.1-8B-Instruct on ShareGPT~\cite{sharegpt} prompts
by sweeping tree depth $\depth \in \{5, 7, 9, 12, 16, 20, 24, 28, 32\}$
at branching factors $\branching \in \{8, 10, 12\}$.

The sweep intentionally spans three regimes predicted by the
micro-benchmark (\Cref{sec:micro}):
\begin{itemize}[nosep,leftmargin=*]
\item \textbf{Loss regime ($\depth \leq 7$, $\numtokens \leq 60$):}
  kernel launch overhead exceeds sparse savings at serving batch sizes.
\item \textbf{Crossover ($\depth \approx 9\text{--}12$, $\numtokens \approx 77\text{--}103$):}
  breakeven shifts with batch size---ragged already wins at
  $\batchsize \geq 32$ here.
\item \textbf{Win regime ($\depth \geq 16$, $\numtokens \geq 137$):}
  ancestor walk dominates; ragged wins at all tested batch sizes.
\end{itemize}

\paragraph{Kernel integration.}
The ragged kernel is injected via a \texttt{torch.matmul}
state-machine hook that intercepts the QK and AV matmuls in EAGLE's
\texttt{LlamaAttention.forward()}.
Unlike \texttt{F.scaled\_dot\_product\_attention}, EAGLE's attention
is implemented as explicit
$Q K^T \to \softmax \to \text{weights} \cdot V$ calls, so patching SDPA
has no effect; the matmul hook is the correct interception point.

\paragraph{Depth--throughput projection.}
The micro-benchmark provides attention-layer speedup $S_{\text{attn}}(\depth)$
across the depth grid.
Applying Amdahl's law (attention fraction $\alpha \approx 0.34$ from profiling):
\begin{equation}\label{eq:amdahl}
  S_{\text{verify}}(\depth) = \frac{1}{(1 - \alpha) + \alpha / S_{\text{attn}}(\depth)}.
\end{equation}

\Cref{tab:e2e_projection} reports projected verify-step speedups
for $\branching{=}10$, $\prefixlen{=}0$ across the depth sweep,
using $S_{\text{attn}}$ values measured on the H100 SXM.

\paragraph{On prefix length.}
At $\prefixlen{=}1024$ (typical mid-generation context),
the micro-benchmark shows $S_{\text{attn}} = 2.82\times$
even at the EAGLE-3 default $(\branching{=}10, \depth{=}7)$ and $\batchsize{=}128$.

\paragraph{Native integration required.}
The current \texttt{torch.matmul} hook fires on \emph{every} matmul
across all 32 layers, incurring Python dispatch overhead that offsets
kernel gains.
Table~\ref{tab:e2e_projection} projects what a zero-overhead native
integration would achieve.
A direct patch to EAGLE's \texttt{LlamaAttention.forward()} or
serving-framework integration (vLLM, SGLang) would realise these numbers
without hook overhead.

\section{Related Work}\label{sec:related}
% =====================================================================

\paragraph{Speculative decoding.}
Leviathan et al.~\cite{leviathan2023fast} and
Chen et al.~\cite{chen2023accelerating} introduced speculative
decoding with linear draft chains. Tree-structured extensions:
Medusa~\cite{cai2024medusa} adds multiple decoding heads;
EAGLE-2/3~\cite{li2024eagle2,li2025eagle3} use
feature-level auto-regression with dynamic trees;
Sequoia~\cite{chen2024sequoia} and SpecInfer~\cite{miao2024specinfer}
use optimal or multi-model tree construction.
All methods share the tree-verification attention step.

\paragraph{Tree-aware attention kernels.}
DeFT~\cite{yao2024deft} is the closest prior work.
DeFT takes a \emph{KV-guided} approach: for each KV position,
it identifies attending queries and distributes work accordingly.
\method{} takes the complementary \emph{Q-guided} approach:
each query walks its own ancestor chain via a closed-form
recurrence (Eq.~\ref{eq:ancestor}).
Q-guided naturally balances work because every query performs
the same $\depth{+}1$ steps, whereas DeFT's root KV is attended
by all $\numtokens$ queries and leaf KV by only 1.

\paragraph{Flash attention.}
FlashAttention~\cite{dao2022flashattention} and
FlashAttention-2~\cite{dao2023flashattention2} tile attention
to fit in SRAM, reducing HBM accesses.
FlashInfer~\cite{ye2025flashinfer} adds ragged batching and
page-table support.
These are \emph{dense}: they process all KV positions regardless
of the mask, which is correct but wasteful for tree-structured masks.
\method{} reuses the online-softmax framework but replaces the
dense KV-block loop with the ancestor walk.

\paragraph{Sparse attention.}
BigBird~\cite{zaheer2020bigbird} and Longformer~\cite{beltagy2020longformer}
exploit \emph{fixed} sparsity patterns (local+global) for long contexts.
Tree-ancestor sparsity is \emph{irregular} and \emph{data-dependent}
(topology varies per request), making block-sparse approaches
inapplicable.

\paragraph{Triton.}
Triton~\cite{tillet2019triton} provides a Python-embedded DSL
for GPU kernels. Our kernel exploits Triton's per-element pointer
arithmetic for scattered ancestor gathers.


% =====================================================================
\section{Discussion}\label{sec:discussion}
% =====================================================================

\paragraph{When is \method{} most beneficial?}
The results reveal a clear regime map:
\begin{itemize}[nosep,leftmargin=*]
\item \textbf{Always wins:} $\prefixlen \geq 1024$ (any $\numtokens$,
  any $\batchsize$). Avoiding weight-matrix materialisation dominates.
\item \textbf{Usually wins:} $\prefixlen{=}0$, $\batchsize \geq 32$.
  Launch overhead is amortised.
\item \textbf{Usually loses:} $\prefixlen{=}0$, $\batchsize \leq 8$,
  $\numtokens < 50$. Dense matmul is faster for small, fixed-size
  computations.
\end{itemize}

\paragraph{EAGLE-3 default operating point.}
EAGLE-3 uses $\branching{=}10$, $\depth{=}7$, $\numtokens \approx 60$.
This is at the boundary: \method{} loses at $\batchsize{=}8$ (0.79$\times$)
but wins at $\batchsize{=}32$ (1.21$\times$) and $\batchsize{=}128$
(1.78$\times$) in tree-only mode.
With $\prefixlen{=}1024$, it wins at \emph{all} batch sizes
(1.76--2.82$\times$).
In production serving with continuous batching and non-trivial
context, \method{} provides consistent improvements.

\paragraph{Why not just use more trees?}
One might argue that EAGLE-3's default trees are small enough
that dense attention is acceptable.
However, the trend in speculative decoding is towards
\emph{larger and deeper trees}~\cite{li2025eagle3,chen2024sequoia},
which improve acceptance rates and tokens-per-step.
\method{}'s advantage grows monotonically with tree size,
making it increasingly relevant as tree budgets expand.

\paragraph{Integration path.}
A native integration into EAGLE-3 or a serving framework
(vLLM~\cite{kwon2023efficient}, SGLang, TensorRT-LLM) requires
replacing the attention forward method directly.
The paged-KV variant (\texttt{ragged\_attention\_paged})
provides the interface needed for PagedAttention.


% =====================================================================
\section{Limitations}\label{sec:limitations}
% =====================================================================

\begin{enumerate}[nosep,leftmargin=*]
\item \textbf{E2E gains require native integration.}
  The current \texttt{torch.matmul} hook adds Python dispatch overhead
  on every matmul across all 32 layers, eroding kernel gains in the
  shallow-tree regime ($\depth \leq 7$, $\numtokens \leq 60$).
  Projected verify-step speedups range from $0.94\times$ (loss, $\depth{=}7$,
  single-user) to $1.38\times$ (win, $\depth{=}32$, $\batchsize{=}128$)
  after removing hook overhead (Eq.~\ref{eq:amdahl}).
  A direct patch to \texttt{LlamaAttention.forward()} is needed.

\item \textbf{Composition, not fusion.}
  The prefix--tree split launches two kernels + a merge.
  A fully fused kernel would eliminate the inter-launch overhead.

\item \textbf{SOTA benchmark dimensions.}
  The SOTA comparison uses $\headdim{=}64$, $\nheads{=}8$ (matching
  FlashInfer/DeFT test parameters).
  The micro-benchmark uses $\headdim{=}128$, $\nheads{=}32$ but
  does not compare against FlashInfer/DeFT at these dimensions.

\item \textbf{Synthetic data.}
  All benchmarks use random Q/K/V tensors. Real weight distributions
  may exhibit different cache behaviour.

\item \textbf{Single GPU.}
  All experiments are on one H100. Multi-GPU tensor-parallel
  settings may shift the bottleneck.

\item \textbf{Fixed tree topology.}
  We assume BFS-ordered $\branching$-ary trees. EAGLE-3's dynamic
  pruning may produce irregular topologies requiring a
  parent-index array extension.

\item \textbf{Small-tree regime loss.}
  At $\numtokens < 50$, $\prefixlen{=}0$, \method{} loses.
  A hybrid dispatch (ragged for large trees, dense for small)
  would be needed in production.
\end{enumerate}


% =====================================================================
\section{Conclusion}\label{sec:conclusion}
% =====================================================================

We presented \method{}, a Triton kernel for tree-structured
speculative decoding that reduces per-head complexity from
$\bigO{\numtokens^2}$ to $\bigO{\numtokens \cdot \depth}$.

Across 1{,}050 micro-benchmark configurations at LLaMA-3.1-8B
dimensions ($\headdim{=}128$, $\nheads{=}32$) on an H100 SXM:
\begin{itemize}[nosep,leftmargin=*]
\item \textbf{Tree-only:} 92\% win rate, median 3.1$\times$, peak 8.9$\times$.
\item \textbf{$\prefixlen{=}1024$:} 100\% win rate,
  median 3.3$\times$, worst case 1.38$\times$.
\item \textbf{$\prefixlen{=}4096$:} 100\% win rate,
  median 4.3$\times$, worst case 1.83$\times$.
\end{itemize}

Against SOTA baselines at large tree sizes:
up to 49$\times$ over FlashAttention-2, 309$\times$ over FlashInfer,
and 114$\times$ over DeFT.

The E2E evaluation (\Cref{tab:e2e_projection}) shows a clear depth-driven
narrative: the kernel loses at shallow trees ($\depth{=}7$, EAGLE-3 default)
but projects $1.21$--$1.48\times$ verify-step improvements at $\depth \geq 16$
via Amdahl's law.
Native integration---replacing the Python matmul hook with a direct attention
patch---is the final step to realise these gains.
We view \method{} as a \emph{kernel-level primitive}
providing the foundation for tree-aware attention in production serving.

\paragraph{Future work.}
(1)~Fused prefix+tree kernel eliminating the merge launch;
(2)~native EAGLE-3 integration replacing the Python hook;
(3)~serving framework integration (vLLM/SGLang) via the paged-KV variant;
(4)~GQA/MQA support for larger models (LLaMA-3.1-70B);
(5)~dynamic tree topologies via parent-index arrays;
(6)~Blackwell SM120 optimisations.
